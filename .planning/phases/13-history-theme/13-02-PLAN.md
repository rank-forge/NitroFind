<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| localStorage → DOM render | Stored query strings (user-authored) are read back and rendered into the history list |
| user → localStorage | Search queries and theme preference are written to origin-scoped localStorage on localhost |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Tampering | renderHistory() history item rendering | mitigate | Render every history item label with `li.textContent = query`, never `innerHTML` — stored query strings are treated as untrusted text (Pitfall 5). Prevents stored-XSS via a query like `<img onerror=...>`. |
| T-13-02 | Denial of Service | localStorage read/write (private mode, quota) | mitigate | Wrap all `localStorage.getItem`/`setItem` in try/catch; degrade silently (no history, no theme persistence) so core search is unaffected (Pitfall 3). |
| T-13-03 | Tampering | localStorage keys | mitigate | Namespace both keys with `nitrofind-` prefix (`nitrofind-history`, `nitrofind-theme`) to avoid cross-app key collision on the origin. |
| T-13-04 | Tampering | JSON.parse of stored history | mitigate | Guard `JSON.parse(localStorage.getItem(...) || '[]')` in try/catch returning `[]` on malformed storage (Pitfall 6) so app init cannot be crashed by corrupted storage. |
| T-13-SC | Tampering | npm/pip/cargo installs | accept | Zero package installs in this phase (RESEARCH Package Legitimacy Audit: none). No supply-chain surface introduced. |

Security assessment: very low risk — local single-user offline tool on `localhost:5000`; localStorage-XSS requires local machine access. textContent + namespaced keys + try/catch are sufficient controls (ASVS L1).
</threat_model>
