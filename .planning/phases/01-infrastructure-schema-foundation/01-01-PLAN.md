<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user-shell → setup_es.py | ES_HOME env var crosses from untrusted shell env into a path used for file writes |
| pip-compile → requirements.txt | Third-party packages and transitive deps enter the project lockfile; hash pinning mitigates supply-chain risk |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | scripts/setup_es.py | mitigate | Validate `ES_HOME` is an absolute existing directory with `os.path.isdir`, and confirm `$ES_HOME/bin/elasticsearch` is a regular file with `os.path.isfile` before any file copy. Path traversal via `ES_HOME=../../etc` is rejected at the isdir/isfile gate. |
| T-01-02 | Tampering | requirements.txt | mitigate | Generate lockfile with `pip-compile --generate-hashes` so every transitive package has a SHA256 hash. Acceptance criterion enforces `--hash=sha256:` presence. Re-installs verify package integrity. |
| T-01-03 | Information Disclosure | config/elasticsearch.yml | mitigate | `network.host: 127.0.0.1` binds ES to loopback — external machines cannot reach the index even on a multi-homed dev host. |
| T-01-04 | Denial of Service | config/jvm.options | accept | Heap pinned at 512 MB; a runaway query could OOM the JVM and crash ES, but the recovery is "restart the app". No external attackers in scope (single-user local tool); risk accepted. |
| T-01-05 | Repudiation | scripts/setup_es.py | accept | No audit log of which config was installed. Single-user local tool; repudiation not in threat model. |
</threat_model>
