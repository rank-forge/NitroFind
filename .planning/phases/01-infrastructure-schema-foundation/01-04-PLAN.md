Execution sequence (must run in this order — PATTERNS.md Shared Patterns + Pitfall 4):
  1. Parse ES_HOME from os.environ.get("ES_HOME"); call validate_es_home() and let ValueError propagate to a top-level handler that prints the message to stderr and sys.exit(1). Do NOT create QApplication first — failing before any Qt object is constructed keeps the negative path test simple.
  2. Create QApplication(sys.argv).
  3. Immediately call apply_stylesheet(app, theme="dark_teal.xml") (PATTERNS.md Shared Patterns — must be before any window).
  4. Construct LoadingWindow.
  5. Construct ESHealthWorker(es_home).
  6. Connect signals BEFORE thread start (Pitfall 4):
       worker.es_ready.connect(on_es_ready)
       worker.es_failed.connect(on_es_failed)
       loading_window.retry_clicked.connect(on_retry_clicked)
       app.aboutToQuit.connect(shutdown_handler)
  7. loading_window.show()
  8. worker.start()
  9. sys.exit(app.exec())

Handler contracts (defined in main.py as module-level closures or methods):
  on_es_ready():
    - client = Elasticsearch("http://localhost:9200")
    - ensure_index(client)                                # SCHEMA-01..04 realized in live index
    - main_window = StubMainWindow(); main_window.show()
    - loading_window.close()
    - Keep main_window referenced from a module-level variable to prevent GC

  on_es_failed(reason: str):
    - Map worker-reason string to Copywriting Contract error text:
        if "exited unexpectedly" in reason: error_text = "Elasticsearch exited unexpectedly. Check your ES_HOME directory and try again."
        else: error_text = "Could not connect to Elasticsearch. Check that ES_HOME is set correctly and try again."
    - Also log the raw `reason` to sys.stderr (UI-SPEC rule: never put raw JVM output in the visible label, but stderr is fine)
    - loading_window.show_error(error_text)

  on_retry_clicked():
    - If worker is still alive and process exists: worker.shutdown_es() (terminates stale process per D-07)
    - worker.wait()                                       # wait for the QThread itself to finish
    - Re-bind a fresh ESHealthWorker(es_home), re-connect its signals (signals are per-instance), re-bind on_retry_clicked to the new instance's emitter
    - loading_window.reset_to_loading()
    - new_worker.start()
    - Update the module-level worker reference so shutdown_handler sees the new instance
    - NOTE: a simpler implementation may rebuild and reassign via a `state` dict or `nonlocal` capture; document the choice in the SUMMARY

  shutdown_handler():
    - if worker is not None: worker.shutdown_es()         # idempotent — D-05 already handles already-exited
    - worker.wait()                                       # ensure QThread.run() returns before the process exits

Entry point:
  if __name__ == "__main__": main()
  main() wraps the execution sequence and the top-level ValueError handler

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ES_HOME env var -> main.py validate_es_home -> ESHealthWorker | Re-asserted from Plan 02 — path-traversal mitigation applies before any subprocess.Popen call |
| ESHealthWorker (background thread) -> main.py handlers (GUI thread) | pyqtSignal cross-thread delivery; Qt serializes safely; reason strings are app-controlled |
| OS process signals (SIGTERM / Ctrl+C / window close) -> shutdown_handler | aboutToQuit fires on all clean exits; shutdown is idempotent (handles already-dead process) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | main.py ES_HOME validation flow | mitigate | validate_es_home is called BEFORE QApplication construction. A malicious ES_HOME cannot cause UI side effects or persistent state. If validation fails, sys.exit(1) is called before any Qt object exists. |
| T-04-02 | Denial of Service | Orphan JVM after crash | mitigate | shutdown_handler is connected to QApplication.aboutToQuit which fires on every clean exit (window close, Cmd+Q, programmatic quit). shutdown_es is idempotent and has a 10s wait + kill fallback. On true crash (segfault of the Python process), the ES JVM does become an orphan — this is the residual risk; mitigated by the manual checkpoint Step 5 verifying no orphan on normal quit. |
| T-04-03 | Information Disclosure | main.py on_es_failed handler | mitigate | Per UI-SPEC Copywriting rules, the user-visible label receives only one of the two static error messages — never the raw worker reason string. The raw reason is written to sys.stderr (developer-only) via logging. |
| T-04-04 | Elevation of Privilege | qt-material apply_stylesheet | accept | qt-material is a trusted PyPI package pinned in requirements.txt with SHA256 hashes (Plan 01 mitigation T-01-02). Risk of malicious stylesheet code is mitigated at supply-chain level by the lockfile. |
| T-04-05 | Repudiation | App startup/shutdown events | accept | No audit log of app starts or shutdowns. Single-user local tool; not in threat model. |
</threat_model>
