<objective>
Add a single REST endpoint ā€” `GET /api/search` ā€” to the existing Flask app in `nitrofind/server.py`. The route is a thin synchronous wrapper over the already-built, Qt-free search stack: parse query params, call `build_filter_clauses()` + `build_search_body()`, call the warm `state["es_client"].search()`, map hits through `ArticleResult.from_es_hit()`, and serialize to the API-01 wire format. To make the route reachable, extend the Phase 6 `state` dict with an `es_client` key and populate it inside `_es_health_poller` immediately before `state["ready"] = True`.

Purpose: Browsers and any HTTP client get a stable JSON search endpoint with ranked results, highlights, and optional facet filters ā€” the backend contract that Phase 8's browser UI will consume.
Output: `api_search()` route + `_result_to_api_dict()` helper + `state["es_client"]` wiring in `nitrofind/server.py`; new `tests/test_search/test_api_search.py` covering API-01, API-02, and the 503 warmup guard.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client ā†’ Flask route | Untrusted query-string params (`q`, `manufacturer`, `era_bucket`, `body_style`, `size`, `from`) cross into the route |
| Flask route ā†’ Elasticsearch | Route-constructed query DSL crosses into the local ES node |

Note: the listener is localhost-only (bound to 127.0.0.1 per Phase 1 / T-06-01), single-user, offline. No authentication is in scope (V2/V4 not applicable per RESEARCH.md Security Domain).

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | `q` param ā†’ ES query | mitigate | `q` is placed only as the string value of `multi_match.query` by `build_function_score_query` ā€” never interpolated as a DSL key or index name. Route passes `q` through `build_search_body()`; no raw-DSL construction in the route. [VERIFIED: query_builder.py] |
| T-07-02 | Tampering | `manufacturer`/`era_bucket`/`body_style` params | mitigate | Values go inside `term` filter value fields via `build_filter_clauses()` ā€” never used as keys or index names. Empty strings coerced to `None` (`request.args.get(...) or None`) so no `{"term": {"field": ""}}` clause is emitted. |
| T-07-03 | Tampering | `index` selection | mitigate | `index="car_articles"` is a hard-coded string literal in the route; never derived from user input (carried from engine.py T-03-04). |
| T-07-04 | Denial of Service | `size` param ā†’ unbounded result set | mitigate | `build_search_body()` clamps `size` to `MAX_RESULT_SIZE=100` and `from_` to non-negative. Route does not bypass `build_search_body`. [VERIFIED: query_builder.py MAX_RESULT_SIZE] |
| T-07-05 | Denial of Service | empty/blank `q` ā†’ ES `BadRequestError` | mitigate | `if not q: return jsonify([])` guard returns before any ES call, preventing the `multi_match` empty-string parse error (Pitfall 3). |
| T-07-06 | Information Disclosure | ES exception detail in 500 response body | accept | `{"detail": str(exc)}` may surface ES internals, but the listener is localhost-only single-user (T-06-01); no remote attacker can reach it. Low risk; aids local debugging. |
| T-07-07 | Denial of Service | search before ES ready ā†’ NoneType `.search` | mitigate | 503 warmup guard (`if not state["ready"]`) runs before any `state["es_client"]` access; client is populated by the single-writer poller before `ready` flips True (Task 1). |

No package installs in this phase ā†’ no supply-chain (T-{phase}-SC) threat applies; RESEARCH.md Package Legitimacy Audit confirms zero new dependencies.

All threats are dispositioned. No high-severity threats remain open.
