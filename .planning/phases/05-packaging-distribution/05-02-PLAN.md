<objective>

hand-written PyInstaller spec file (nitrofind.spec) and a post-build assembly
script (scripts/build_dist.py). Together they turn the repo into a distributable
zip archive that a first-time user can extract and run on a clean Windows machine
with no Python or Java installed.

reads to freeze the application, and the assembly script that copies the Elasticsearch
directory alongside the frozen bundle and zips everything for distribution. A human
checkpoint gates on a manual clean-machine smoke test — the only verification that
cannot be automated.

Output:
  - nitrofind.spec (project root)
  - scripts/build_dist.py
  - Manual verification: distribution archive extracts and launches on a clean Windows
    machine, reaching ES health-check pass and MainWindow visible state.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| build machine → PyInstaller | pyinstaller.exe executes spec file (Python code) — spec is version-controlled and developer-authored; no untrusted input |
| build script → ES_BUNDLE path | build_dist.py reads ES_BUNDLE env var and copies a directory; validates isdir before acting |
| distribution zip → end-user machine | user extracts zip to a location they control; zip slip risk is on the extraction step (user-controlled) |
| NitroFind.exe → elasticsearch.bat subprocess | _start_process spawns the bundled ES binary (already guarded by Plan 01's DEVNULL + validate_es_home) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Denial of Service | ESHealthWorker._start_process windowed frozen mode | mitigate | Covered by Plan 01: stdin/stdout/stderr=DEVNULL, close_fds=True. nitrofind.spec sets console=False which triggers this failure mode — the fix must be in place before this build runs. Tested by test_subprocess_handles.py (Plan 01). |
| T-05-02 | Tampering | inject_es_config / validate_es_home call ordering | mitigate | Covered by Plan 01: inject_es_config wired in main.py before validate_es_home. nitrofind.spec bundles config/ as datas so sys._MEIPASS/config/elasticsearch.yml is present at runtime. If datas entry is missing, inject_es_config raises OSError (caught + logged); ES then starts with default security; cluster health probe fails with SSL error — error is surfaced in LoadingWindow, not silently swallowed. |
| T-05-03 | Information Disclosure | ES_HOME env var relied on in frozen build | accept | Covered by Plan 01: resolve_es_home() uses sys.executable-relative glob in frozen mode. ES_HOME env var is never read in the frozen distribution path. Accepted because Plan 01 eliminates the threat. |
| T-05-04 | Tampering | Zip slip — user-extracted archive path traversal | transfer | build_dist.py constructs archive paths as Path("NitroFind") / f.relative_to(DIST_DIR) — all entries are rooted under NitroFind/ (no absolute paths, no .. components). Extraction is user-controlled. Document "extract to a trusted local directory" in the smoke-test instructions. |
| T-05-SC | Tampering | pip install pyinstaller (build-time only) | mitigate | Per RESEARCH.md Package Legitimacy Audit: pyinstaller [OK] — 13+ years, ~9.8M/week downloads, github.com/pyinstaller/pyinstaller. Build-time only; never shipped. No human-verify checkpoint required (Approved disposition). |
</threat_model>
