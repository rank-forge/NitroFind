### New components (Phase 12)

#### `.pagination-row` — Pagination control row

Placed after `#results-list`, inside `.results-view`. Always rendered in the DOM; always visible when `data-state="results"`. Visibility is controlled by the parent state machine (`.results-view` display rule), not by the pagination row itself.

**HTML structure:**
```html
<div class="pagination-row" id="pagination-row">
  <button type="button" id="prev-btn" disabled>&#8592; Previous</button>
  <button type="button" id="next-btn">Next &#8594;</button>
</div>
```

**CSS rules (add to `static/css/style.css`):**
```css
.pagination-row {
  display: flex;
  gap: 0.5rem;
  padding: 0.75rem 1.5rem;
  justify-content: center;
}

.pagination-row button {
  padding: 0.375rem 0.875rem;
  font-size: 0.85rem;
  background-color: var(--bg-input);
  color: var(--text-primary);
  border: 1px solid var(--border);
  border-radius: var(--radius);
  cursor: pointer;
  transition: border-color var(--transition), background-color var(--transition);
}

.pagination-row button:hover:not(:disabled) {
  border-color: var(--text-secondary);
}

.pagination-row button:disabled {
  opacity: 0.4;
  cursor: not-allowed;
}
```

Source: RESEARCH.md Pattern 9 — mirrors `.sort-btn` token references exactly; no raw hex values.

## Interaction Contract

### Pagination button states

| State | `prev-btn` | `next-btn` |
|-------|-----------|-----------|
| Page 1, single page (total ≤ 10) | disabled | disabled |
| Page 1, multiple pages (total > 10) | disabled | enabled |
| Middle page | enabled | enabled |
| Last page (`page * 10 >= total`) | enabled | disabled |

**Disabled presentation:** `opacity: 0.4`, `cursor: not-allowed`. Buttons remain visible at all times — no hiding/toggling `display`. Removing them from the DOM causes layout shift.

**Click behavior:**
- `prev-btn`: `currentPage -= 1; runSearch(currentQuery)` — guarded by `currentPage > 1` check
- `next-btn`: `currentPage += 1; runSearch(currentQuery)`

**Page reset triggers:** `currentPage` resets to 1 on:
1. New query typed in either search input (`handleSearchInput`)
2. Any filter change (`onFilterChange`)
3. Any sort change (`onSortChange`)

`currentPage` does NOT reset on: Escape key, back-button navigation to results view.

### Stats line update (PAGE-02)

| Before Phase 12 | After Phase 12 |
|----------------|----------------|
| `"10 results (0.08s)"` — page count, not total | `"248 results (0.08s)"` — total across all pages |
| `results[0].took_ms` per-item | `data.took_ms` from wrapper |

Format: `"{total} results ({took_ms / 1000 to 2dp}s)"`
Zero state: `"No results"` (unchanged string)

### Keyboard navigation (unchanged)

ArrowDown/ArrowUp/Enter operate on `currentResults` (current page items only, indices 0–9). No change needed — `currentResults` holds the current page's items after Phase 12 just as before.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Previous button | `← Previous` (HTML entity `&#8592;` + space + text) |
| Next button | `Next →` (text + space + HTML entity `&#8594;`) |
| Stats line — results present | `{N} results ({T}s)` — no page number, no "Page X of Y" |
| Stats line — no results | `No results` (unchanged existing string) |
| Empty state (no query) | Not shown — home view handles this; no empty-state needed for pagination |
| Error state | No visible error state — search failures are silent (`console.error`); unchanged from existing behavior |

**Destructive actions:** None in Phase 12.

Source: ROADMAP.md Phase 12 success criteria item 3 — `"248 results (0.08s)"` is the exact prescribed format. No "Page X of Y" decoration. Source confirmed from RESEARCH.md Pattern 7 note: "The success criterion says '248 results (0.08s)' — it does NOT include a page number."
