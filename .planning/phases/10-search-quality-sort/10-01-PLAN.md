<objective>
Add typo tolerance, phrase routing, and sort-by-field support to the backend search layer. This is the backend half of Phase 10: all logic lives in `query_builder.py` and the Flask `api_search` route, fully covered by unit tests against existing pytest files.

Purpose: Users who misspell manufacturer names ("Ferari", "Lamborgini") must still get correct results (QURY-01); users who wrap a phrase in quotes ("V8 engine") must get exact-phrase matches (QURY-02); the API must accept an optional `sort` param mapping to ES field sorts (SORT-02).

Output: Modified `query_builder.py` (phrase-branching base query, `_build_sort_clauses` helper, `sort` param on `build_search_body`), modified `server.py` (`_VALID_SORTS` allowlist, `sort` kwarg on `client.search()`), and new unit tests in both existing test files.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-SORT | Tampering | `sort` param in `api_search` | mitigate | Allowlist `_VALID_SORTS = {"relevance","date","size"}`; any other value (incl. JSON/script blobs like `sort={"script":...}`) coerced to `None` before reaching `build_search_body` — `_build_sort_clauses` only ever emits hard-coded field-sort dicts, never user text |
| T-10-PHRASE | Tampering / DoS | phrase detection in `build_function_score_query` | mitigate | `len(query_text) > 2` guard rejects empty `""` so ES never receives a zero-length phrase that 400s; stripped phrase text goes into `multi_match.query` VALUE only, never a DSL key (inherits T-07-01) |
| T-10-FUZZ | DoS | fuzzy `multi_match` expansion | accept | ES default `max_expansions=50` is acceptable for a single-user local index; `prefix_length=1` further bounds expansion — no override needed |
| T-10-01 | Information Disclosure | new test fixtures | accept | Tests use synthetic mock ES responses; no real data or secrets |
</threat_model>
